# R9.DefinitionsFirst

**Spec:** For any statutory chapter or regulatory subchapter under analysis, the chapter's Definitions section (whatever it is named — "Definitions", "Definitions and construction", "As used in this chapter", etc.) must be retrieved and consumed before any operative provision of that chapter is analyzed. Plans that analyze operative sections without first retrieving the definitions are rejected and amended.
**Enforcement point:** The orchestrator's `before_agent_run` hook validates the dispatched plan: for every operative-provision retrieval in the plan, the corresponding Definitions section must appear earlier in the retrieval sequence (or be marked `definitions_not_applicable: <reason>`). Plans failing this check are amended automatically.
**Severity:** blocking.
**Failure mode:** Plan is rejected with `DEFINITIONS_FIRST_VIOLATION: operative section ca-lab-2775 will be analyzed before the chapter's Definitions section is retrieved. Inserting Definitions retrieval at plan position 0.` The plan is amended in place; the analysis re-dispatches. If the chapter's Definitions section cannot be located (e.g., the chapter doesn't have a separate definitions block — some Labor Code articles inline their definitions), the orchestrator marks the requirement `not_applicable` with a recorded reason and proceeds.

## Detail

R9 is the procedural rule that prevents the most common subtle error in statutory interpretation: applying everyday meaning to a defined term. The Labor Code's "employer" is not the dictionary "employer." The Insurance Code's "household member" is not the everyday "household member." Definitions sections exist specifically because the legislature has bound the operative-provision text to special meanings, and reading the operative provisions without those bindings produces conclusions that look right but are legally wrong.

The PRD §6 Methodology Part II explicitly lists this as step 1 of the 7-step research sequence per state: "Locate and read the Definitions section first." R9 is the rule that enforces this procedurally.

**Mechanism.** Every plan dispatched to a sub-agent is structured as an ordered list of retrieval-and-analyze steps. The plan validator inspects each operative-provision step and verifies that earlier in the sequence there is a retrieval of the corresponding chapter's Definitions section. For California Labor Code, the chapter's definitions are typically at the start of each Division/Part/Chapter (e.g., Lab. Code §§ 200–211 for Division 2, Part 1, Chapter 1). The validator uses the corpus's chapter-structure metadata to identify the right section.

**When definitions are inline.** Not every chapter has a separate Definitions section. Some chapters inline their definitions in the operative provisions (e.g., "for purposes of this subdivision, 'X' means..." inside § 2775(d)). In those cases, the chapter-structure metadata in the corpus marks the chapter as `definitions_inline`, and R9 is satisfied as long as the inline-definition section is retrieved. The validator handles this automatically — no Drafter action required.

**When definitions reference other codes.** Sometimes the Definitions section says "as defined in Government Code section X." That cross-reference must itself be traced — R8 handles this. R9 only requires that the *chapter's* Definitions section be retrieved; R8 chases any cross-references that section contains.

**Multi-chapter analysis.** When the analysis spans multiple chapters (e.g., Labor Code Division 2, Part 1 + Unemployment Insurance Code Division 1, Part 1), R9 requires the Definitions section of *each* chapter to be retrieved. The plan validator enforces this per chapter; missing any one of them blocks the plan.

**Why this is its own rule.** R9 could conceptually be folded into R15 (CompletenessGate) — the completeness check already requires definitions to have been retrieved. But R9 fires at plan-construction time, which is earlier than R15. Catching the omission at plan time prevents wasted retrievals and Drafter cycles on chapters whose definitions weren't even consulted. R15 is the backstop; R9 is the entry gate.

## Interactions with other rules

- **R8 CrossRefTrace** handles cross-references *within* the Definitions section once retrieved (definitions often reference other codes' definitions). R9 ensures Definitions is retrieved; R8 handles what it contains.
- **R15 CompletenessGate** is the backstop. If R9 somehow misses (or is overridden), R15's "definitions" completeness criterion catches it at the end.
- **R3 NoMemoryStatutes** ensures the Drafter doesn't reach for "well-known definitions" from training memory — if the chapter's Definitions section isn't in retrieval_log, the Drafter can't define a term in the deliverable.

## Test cases

✅ Compliant: chapter definitions retrieved first
```
TaskSpec analyzes Cal. Lab. Code § 2775 (operative). Chapter: Division 3, Part 1, Chapter 2.
Plan as drafted by orchestrator:
1. Retrieve ca-lab-2775 (Definitions chapter is the section block 2750-2773 per corpus metadata).
2. Retrieve ca-lab-2750..2773 (Definitions block).
3. Analyze ca-lab-2775 against the definitions.
R9 validator: definitions block 2750-2773 is retrieved at step 2, before step 3 analyzes 2775. PASS.
```

❌ Non-compliant: operative before definitions
```
Plan as drafted:
1. Retrieve ca-lab-2775.
2. Analyze ca-lab-2775.
3. (No definitions retrieval.)
R9 validator: { block: true, reason: "DEFINITIONS_FIRST_VIOLATION: operative section ca-lab-2775 will be analyzed at step 2 before chapter's Definitions section (ca-lab-2750..2773 per corpus metadata) is retrieved. Inserting Definitions retrieval at step 1." }
Plan amended; re-dispatched.
```

✅ Compliant: definitions are inline
```
Chapter has no separate Definitions section. Corpus metadata marks chapter as definitions_inline.
Plan retrieves operative section and a flag-marked inline-definitions paragraph from the same section.
R9 validator: definitions_inline is true; the inline-definitions retrieval satisfies the requirement. PASS.
```

❌ Non-compliant: multi-chapter analysis, one chapter's definitions missing
```
Analysis spans Cal. Lab. Code Chapter 2 (Definitions: 2750-2773) AND Cal. Unemp. Ins. Code Chapter 1 (Definitions: 100-145).
Plan retrieves Lab. Code definitions but not Unemp. Ins. Code definitions before analyzing Unemp. Ins. Code provisions.
R9 validator: { block: true, reason: "DEFINITIONS_FIRST_VIOLATION: Unemp. Ins. Code definitions (100-145) not retrieved before analyzing Unemp. Ins. Code § 621. Inserting at next plan step." }
```

⚠️ Edge case: definitions section not found in corpus
```
Plan tries to analyze a section in an obscure chapter; corpus metadata indicates no Definitions section exists for the chapter.
R9 validator: marks definitions_not_applicable with reason "chapter X has no Definitions section per corpus structure metadata". Analysis proceeds; R15 records the absence in the completeness block.
```

✅ Compliant: definitions also retrieved by chain from R7 or R8
```
R7 added a regulation retrieval for an implementing reg.
The regulation references a definition in the parent statute's Definitions section.
R8 traced the cross-ref and retrieved the Definitions section.
R9's check at plan-validate time sees Definitions retrieved (via R8 chain) before any analysis of operative provisions. PASS.
```

## Open questions

- **Chapter-structure metadata source.** Per-jurisdiction chapter structure (where the definitions live in each chapter) is metadata that has to be captured at corpus ingest time. The Phase 2 ingestion-worker code (`apps/ingestion-worker/`) needs to extract chapter boundaries and definition-block markers from the official source layout. For CA, leginfo.legislature.ca.gov has consistent structure; for other states, parsing reliability varies. Phase 2 deliverable.
- **Subsection-level definitions.** A subdivision may define a term used only within that subdivision (`for purposes of this subdivision, "X" means...`). R9 doesn't enforce subdivision-level retrieval; it operates at chapter granularity. Whether to extend to subdivision granularity is a Phase 4 decision based on observed false-negative rate.
- **Override for known-stable definitions.** Some terms have widely-accepted, jurisdiction-stable definitions (e.g., "wages" in CA Lab. Code). Should the system permit caching a once-retrieved Definitions section across multiple TaskSpecs to avoid re-retrieval? Defer to Phase 4; v1 retrieves fresh per TaskSpec.
