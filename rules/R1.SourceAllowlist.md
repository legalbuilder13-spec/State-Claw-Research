# R1.SourceAllowlist

**Spec:** Every KB tool call (any tool name matching `kb_*`) must validate that the tool's declared source identifier is present in the active source profile's allowlist. The allowlist is the union of all `*_source` and `*_sources` keys in the YAML at `source-profiles/profile-{a,b}.yaml` for the profile named in the active TaskSpec.

**Enforcement point:** The `before_tool_call` hook in the legal-research plugin, fired per KB tool invocation. The hook reads the active TaskSpec's `source_profile` field (`"A"` or `"B"`), loads the corresponding profile YAML, computes the allowlist, and rejects any tool whose source is not in the allowlist.

**Severity:** blocking.

**Failure mode:** The tool call is blocked before it executes. The model receives an error tool result with the exact text:

```
SOURCE_NOT_ALLOWED: tool "<tool_name>" targets source "<source_id>" which is not in the active source profile (<profile_id>) allowlist. Allowlisted sources: [<list>]. Either switch source profile or pick a different tool.
```

The block is logged to the audit chain with the rule ID, the failing tool name, the failing source, the active profile, and the full allowlist at decision time. The Drafter sub-agent must not cite content that was never retrieved — R3 catches downstream attempts to do so; R1 prevents the retrieval entirely.

## Detail

The source allowlist is the system's first-line defense against hallucination by citation. Every citation in the final deliverable must trace to a span retrieved this session via an allowed tool from an allowed source. If we never retrieved it, R3 prevents citing it; if a source isn't on the allowlist for the active profile, R1 prevents retrieving from it in the first place.

**Why an allowlist rather than a blocklist.** Closed-world retrieval is one of the PRD's load-bearing design principles (§3 goal 2). A blocklist invites silent expansion of citation sources as new tools are added; an allowlist forces every new source to be explicitly enrolled in a source profile YAML and code-reviewed.

**Per-profile allowlists.** Profile B (Lexis-free, the default per PRD §1) allowlists local ingested corpora and free citator chains:

- `local_corpus_state_official` — statutes ingested from official state legislative websites
- `local_corpus_state_admin` — administrative-code corpus
- `courtlistener_api` — primary case source
- `state_legislative_official_site`, `justia`, `cornell_lii`, `openstates_api` — secondary statute sources
- `cap_api`, `google_scholar` — secondary case sources
- `casetext_free`, `vlex_fastcase` — opt-in free citators

Profile A (Lexis-enabled, opt-in per PRD §8) extends with:
- `lexis_brief_analysis`
- `lexis_shepards`
- `lexis_doc_get`

The exact YAML key names are authored in `source-profiles/profile-{a,b}.yaml` and are the single source of truth — R1's implementation reads them at runtime, not at compile time, so adding a new source to a profile takes effect on the next TaskSpec without code changes.

**Tools with no source.** Some KB tools are source-agnostic — e.g., `kb_span_get_by_hash` returns a span that was previously retrieved-and-logged this session. These don't have a "target source" to check because the source was already validated when the span was originally fetched. R1's implementation lists these tools explicitly (in `extensions/agentic-law-os/rules/r1-config.json` or similar) and skips them.

**Jurisdiction-awareness.** R1 is jurisdiction-aware as a defense-in-depth pairing with R2: `kb_statutes_search(jurisdiction="CA", …)` requires both (a) `local_corpus_state_official` be in the active profile's allowlist AND (b) CA be in the active TaskSpec's `jurisdictions` list AND (c) the CA corpus be ingested (per `corpus_freshness` table). If any of these fails, R1 rejects with a precise reason rather than letting the call partially succeed.

**Why R1 is not a regex on URLs.** An earlier draft of this rule was specified as "block any URL not matching an allowlisted domain regex." That approach failed two ways: (1) some allowed sources (e.g., `local_corpus_state_official`) are not URLs at all, they're database queries; (2) URL regexes are easy to circumvent with proxies, redirects, or CDN aliases. The current spec — every tool declares its source identifier and the allowlist is identifier-based — is impossible to circumvent by URL manipulation.

## Interactions with other rules

- **R2 JurisdictionLock** is the per-jurisdiction analog: R1 controls *which sources* are reachable; R2 controls *which jurisdictions* are addressable. Both fire at `before_tool_call`.
- **R14 NoLawFirmCitation** is the render-time analog: R1 prevents fetching from disallowed sources; R14 prevents citing disallowed source *categories* (law-firm blogs, news, etc.) even if a fetch slipped through.
- **R3 NoMemoryStatutes** is downstream: if R1 fails to block a fetch (bug), R3 still catches an unanchored citation.
- **W1 PartialAcrossStates** depends on R1: if a multi-state survey blocks a state's fetches because the allowlist excludes a jurisdiction-specific source, W1 surfaces the partial-coverage warning instead of silently omitting that state.

## Test cases

✅ Compliant: in-allowlist statute retrieval
```
TaskSpec: { jurisdictions: ["CA"], source_profile: "B" }
Tool call: kb_statutes_search({ jurisdiction: "CA", query: "independent contractor" })
Result: tool executes; returns ranked CA Labor Code chunks; entries logged to retrieval_log.
```

✅ Compliant: source-agnostic tool
```
TaskSpec: { jurisdictions: ["CA"], source_profile: "B" }
Tool call: kb_span_get_by_hash({ hash: "sha256:..." })
Result: tool executes (R1 skips source-agnostic tools); returns the previously-retrieved span if it exists in retrieval_log, otherwise NOT_FOUND.
```

❌ Non-compliant: Lexis tool under Profile B
```
TaskSpec: { jurisdictions: ["CA"], source_profile: "B" }
Tool call: kb_lexis_brief_analysis({ query: "..." })
Result: { block: true, reason: "SOURCE_NOT_ALLOWED: tool \"kb_lexis_brief_analysis\" targets source \"lexis_brief_analysis\" which is not in the active source profile (B) allowlist. Allowlisted sources: [local_corpus_state_official, local_corpus_state_admin, courtlistener_api, ...]. Either switch source profile or pick a different tool." }
```

❌ Non-compliant: web fetch outside allowlist
```
TaskSpec: { jurisdictions: ["CA"], source_profile: "B" }
Tool call: kb_web_fetch({ url: "https://www.somelawfirm.com/blog/ab5-explained" })
Result: { block: true, reason: "SOURCE_NOT_ALLOWED: domain \"somelawfirm.com\" is not in the active source profile (B) allowlist. (Note: law-firm blogs are orientation-only and not citable per R14 even when accessible.)" }
```

❌ Non-compliant: source-agnostic-disguised-as-keyword
```
TaskSpec: { jurisdictions: ["CA"], source_profile: "B" }
Tool call: kb_statutes_search({ jurisdiction: "CA", query: "verbatim text from a private database I'm copy-pasting", source_hint: "private_db" })
Result: tool executes — the source_hint param is ignored by R1 because the tool name maps to local_corpus_state_official. The model gets a normal corpus search response with no acknowledgement of the hint. (This test exists to confirm that R1 cannot be steered by free-text params.)
```

❌ Non-compliant: profile not loaded
```
TaskSpec: { jurisdictions: ["CA"], source_profile: "C" }   // typo / unknown
Tool call: kb_statutes_search({ jurisdiction: "CA", query: "..." })
Result: { block: true, reason: "SOURCE_NOT_ALLOWED: active source profile \"C\" not found in source-profiles/. Valid profiles: A, B." }
// (R1 fail-closed: missing profile → block all fetches, not allow all.)
```

❌ Non-compliant: jurisdiction not ingested
```
TaskSpec: { jurisdictions: ["WY"], source_profile: "B" }   // WY not yet ingested
Tool call: kb_statutes_search({ jurisdiction: "WY", query: "..." })
Result: { block: true, reason: "SOURCE_NOT_ALLOWED: local_corpus_state_official has no entries for jurisdiction \"WY\" (corpus_freshness check). Ingest WY before researching it, or remove WY from TaskSpec.jurisdictions." }
```

## Open questions

- **Orientation reads.** PRD §3 distinguishes "closed-world retrieval" from "law-firm blogs and vendor sites are orientation only, never citation sources." Interpretation A: orientation sources are NOT in the source-profile allowlist (R1 blocks them entirely, v1 default). Interpretation B: orientation sources are in a separate `orientation_only:` block in the profile YAML, R1 allows fetch, R14 blocks citation. Recommend Interpretation A for v1 (simpler, stricter, "refuse rather than fabricate"). Revisit if attorneys complain they can't access orientation material for context.
- **Allowlist hot-reload.** Should a profile YAML change take effect within an active TaskSpec or only on the next TaskSpec? Current spec is "on the next TaskSpec" (the profile is loaded once at TaskSpec creation and cached for the run). This avoids mid-run drift but means a profile fix requires re-running the escalation. Acceptable for v1.
- **Per-tool source declaration source of truth.** Should the tool's source identifier be declared in the tool's manifest (`extensions/agentic-law-os/openclaw.plugin.json`) or computed from the tool name (`kb_lexis_*` → `lexis_*`)? Manifest is more explicit and survives renames; computed is less maintenance. Decide in Phase 4 when the plugin pack is being built.
