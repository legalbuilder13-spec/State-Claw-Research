# R5.HashEcho

**Spec:** The Verifier sub-agent independently re-fetches every anchored span referenced in the draft (via `kb_span_get_by_hash`) and exact-substring-matches the asserted quoted text against the returned content. Any material discrepancy — missing hash, hash collision against a different span, or asserted text not present as a contiguous substring of the returned span — blocks delivery.
**Enforcement point:** The Verifier sub-agent runs as a separate session (`api.runtime.subagent.run({ session: "agent:main:subagent:verifier", ... })`) after the Drafter completes and before the Renderer fires; failures are reported to the orchestrator via the standard sub-agent result envelope and surface as `before_agent_finalize` blocks.
**Severity:** blocking.
**Failure mode:** The Verifier emits a structured `verifier-output` artifact (per `schemas/verifier-output.schema.json`) listing every anchor with status `pass | hash_not_found | text_mismatch | corpus_drift`. Any non-pass status triggers an `action: "revise"` with instructions naming the failing anchor and the exact discrepancy. After 2 revise loops with persistent failures, R11 LowConfidenceGate fires.

## Detail

R5 is the independent re-verification of every anchor produced by the Drafter. Where R4 ensures the Drafter *can't write* an unanchored quote, R5 ensures that anchors the Drafter *did write* actually match the source. This is defense in depth: even if the Drafter satisfies R4's structural requirement (every quote has an anchor), the anchor itself could be wrong (hash forged, char offsets off, text drifted since fetch). R5 catches that.

**Why a separate sub-agent.** The Verifier is implemented as a sub-agent rather than a static validator for two reasons:

1. *Independence.* The Verifier runs in a fresh session with its own model context. It doesn't share the Drafter's prompt history, so it can't be steered into agreeing with the Drafter. It only sees the structured anchor list + access to `kb_span_get_by_hash`.
2. *Composition.* Some anchors require interpretation (e.g., a quote that spans a sentence boundary, or a quote with a footnote marker in the middle). A static validator can't handle these; a sub-agent with the source text in front of it can.

**The four statuses.**

- `pass`: the anchor's `(doc_id, char_start, char_end, hash)` resolves to a real span in `retrieval_log`, the hash matches the live computed hash of that span, and the Drafter's asserted quoted text is a contiguous substring of the span.
- `hash_not_found`: the hash isn't in `retrieval_log` (forged anchor or stale corpus reference).
- `text_mismatch`: the hash resolves to a span, but the Drafter's asserted quoted text is not a contiguous substring of that span. This is the most dangerous failure — it means the Drafter wrote down quoted text that doesn't appear in the source.
- `corpus_drift`: the hash exists but doesn't match the freshly-computed hash of the same `(doc_id, char_start, char_end)`. This means the corpus changed between Drafter time and Verifier time (re-ingestion mid-session). Handled by R10 CurrencyTag.

**Material discrepancy.** Not every discrepancy is material. Whitespace normalization (collapsing runs of spaces, normalizing newlines) and Unicode normalization (NFC vs. NFD) are NOT material — the Verifier normalizes both the asserted text and the source span before comparison. Capitalization, punctuation, and word order ARE material. If the Drafter wrote `"the worker"` but the source says `"a worker"`, that's a material discrepancy and R5 blocks. (This level of strictness is intentional — legal text is read for every word.)

**Speed.** Verifying every anchor in a typical Slack-response deliverable (5–15 anchors) takes ~2–5 seconds via `kb_span_get_by_hash` (a single keyed lookup per anchor, no embeddings, no LLM calls except the Verifier sub-agent's structured-output emit). The deliverable's total latency budget allows for this — see PRD §19 acceptance criterion 2 ("response in <2 minutes").

**Failure batching.** When multiple anchors fail in the same draft, the Verifier reports all failures in one artifact (not one-at-a-time). The Drafter's revise loop sees the full list and can fix all of them in one revision rather than ping-ponging. This is also what makes R5 + R11 work cleanly: 2 revise loops is plenty if each loop addresses all failures at once.

## Interactions with other rules

- **R3 NoMemoryStatutes** ensures every assertion has a cite to a retrieved doc_id; R5 ensures every anchor *within* those cites actually matches the source. R3 catches existence; R5 catches fidelity.
- **R4 QuotingRule** is the Drafter-side structural requirement that anchors must exist; R5 is the Verifier-side semantic requirement that anchors must match.
- **R10 CurrencyTag** handles the `corpus_drift` case — if the Verifier sees a stale hash that no longer matches the live span, R10 fires alongside R5 and the orchestrator decides whether to refresh and re-draft.
- **R11 LowConfidenceGate** is the escalation path after 2 failed revise loops.

## Test cases

✅ Compliant: clean anchor with matching span
```
Drafter emits a quote with anchor: { doc_id: "ca-lab-2775", char_start: 0, char_end: 90, hash: "sha256:abcd..." }
Asserted quoted text: "a person providing labor or services for remuneration shall be considered an employee"

Verifier:
1. kb_span_get_by_hash("sha256:abcd...") returns span text "a person providing labor or services for remuneration shall be considered an employee" (chars 0-90 of ca-lab-2775).
2. Live re-hash of the span: "sha256:abcd..." — matches.
3. Asserted text exact-matches the returned span.
Result: { status: "pass" }
```

❌ Non-compliant: text_mismatch
```
Drafter emits quote with valid anchor pointing to chars 0-90 of ca-lab-2775.
Asserted quoted text: "the worker shall be classified as an employee"

Verifier:
1. kb_span_get_by_hash succeeds; span text is "a person providing labor or services for remuneration shall be considered an employee".
2. Asserted text "the worker shall be classified as an employee" is NOT a contiguous substring of the returned span.
Result: { status: "text_mismatch", reason: "R5: asserted quote not present in anchored span. Anchored doc=ca-lab-2775 chars 0-90 reads: \"a person providing labor or services for remuneration shall be considered an employee\". Drafter asserted: \"the worker shall be classified as an employee\". The Drafter is either paraphrasing (rewrite without quotation marks) or wrote down text that does not appear in the source." }
```

❌ Non-compliant: hash_not_found
```
Drafter emits quote with anchor hash: "sha256:forged123..."

Verifier:
1. kb_span_get_by_hash("sha256:forged123...") returns NOT_FOUND.
Result: { status: "hash_not_found", reason: "R5: anchored hash sha256:forged123... not present in retrieval_log. Either the span was never retrieved this session (R3 violation should have caught this upstream) or the hash was fabricated." }
```

⚠️ Conditional: corpus_drift
```
Drafter emits quote with anchor hash sha256:abcd... (retrieved 30 minutes ago).
Mid-session, the ingestion worker re-ingested ca-lab-2775 (new amendment text).

Verifier:
1. kb_span_get_by_hash("sha256:abcd...") returns the OLD span (retrieval_log preserves historical retrievals by hash).
2. Live re-hash of ca-lab-2775 chars 0-90 returns sha256:zzz... — different from the anchored hash.
Result: { status: "corpus_drift", reason: "R5+R10: anchor's hash matches the historical retrieval, but the live source has changed (re-ingestion at <timestamp>). Orchestrator decision: (a) if the live text is semantically identical (whitespace only): allow and log W2; (b) if the live text is materially different: trigger R10 refresh and re-Draft the affected claim." }
```

❌ Non-compliant: char offsets outside doc bounds
```
Drafter emits anchor: { doc_id: "ca-lab-2775", char_start: 50000, char_end: 50100, hash: "sha256:..." }
(but ca-lab-2775 is only 20,000 chars long)

Verifier:
1. kb_span_get_by_hash returns NOT_FOUND or RANGE_ERROR.
Result: { status: "hash_not_found", reason: "R5: anchor char_start (50000) exceeds doc_id ca-lab-2775 length (20000). Anchor is invalid; either the offsets are wrong or doc_id is wrong." }
```

## Open questions

- **Verifier model selection.** The Verifier sub-agent's only job is "given a span and an asserted quote, do they match exactly modulo normalization?" — this could be a small model (or a non-LLM deterministic check). Initial implementation uses the same model as the Drafter for simplicity; Phase 4 should benchmark a Haiku-class model for cost savings.
- **Normalization aggressiveness.** Current rule: whitespace and Unicode normalization only. Should we also normalize across smart-quote variants ("…" vs. "…"), em-dash vs. hyphen, and ligatures (fi vs. fi)? Tighter normalization reduces false positives at the cost of permitting subtler drift. Recommend tighter normalization with explicit logging when any normalization is applied so a reviewer can audit.
- **Batched re-fetch performance.** If a deliverable has 20+ anchors, the Verifier makes 20+ kb_span_get_by_hash calls. Batching to a single `kb_span_get_by_hashes(["sha256:a", "sha256:b", ...])` would be faster — add this tool in Phase 3 if the latency becomes noticeable.
