# R2.JurisdictionLock

**Spec:** Every KB tool call must carry a `jurisdiction` parameter (or derive one from session context) that is present in the active TaskSpec's `jurisdictions[]` array. Tools targeting any jurisdiction outside that list are blocked before execution.
**Enforcement point:** Tool parameter schema (TypeBox enum keyed to TaskSpec.jurisdictions) plus the `before_tool_call` hook as a defense-in-depth check.
**Severity:** blocking.
**Failure mode:** The tool call is blocked with the error text `JURISDICTION_VALIDATION_ERROR: tool "<tool_name>" targets jurisdiction "<requested>" which is not in TaskSpec.jurisdictions [<list>]. Either expand TaskSpec.jurisdictions at intake or pick a different tool.` The block is logged to the audit chain with the active TaskSpec ID, the failing tool, the requested jurisdiction, and the locked jurisdiction list.

## Detail

R2 is the per-jurisdiction analog of R1's per-source allowlist. Where R1 controls *which corpora* are reachable, R2 controls *which states* the agent may speak about in this run. The two rules fire at the same hook (`before_tool_call`) and are deliberately redundant — a bug or misconfiguration in one is still caught by the other.

**Why a hard lock at intake.** The PRD §6 Methodology Part I treats jurisdiction selection as a Tier-1 intake decision: the operator names the jurisdictions when filing the escalation, the orchestrator freezes those into the TaskSpec, and nothing downstream can broaden the scope. This is what prevents the common failure mode where an attorney asks about California and the agent helpfully adds "and in Illinois, the rule is…" — useful in casual conversation, but dangerous in a deliverable that will be quoted in counsel review.

**TaskSpec.jurisdictions is the single source of truth.** The schema at `schemas/task-spec.schema.json` defines `jurisdictions` as a non-empty array of ISO-3166-2 codes (`"US-CA"`, `"US-IL"`, etc.) with optional sub-jurisdiction qualifiers (e.g., `"US-CA:los-angeles-county"`). The legal-research plugin reads this once at TaskSpec construction and binds it into every tool's parameter schema for the run, so the TypeBox `Type.Enum` on each KB tool's `jurisdiction` param is the *exact* enumeration of valid values for that run. A model that tries to call `kb_statutes_search({ jurisdiction: "TX", ... })` when the TaskSpec is `["US-CA"]` fails schema validation before R2 even fires — the param value isn't in the enum.

**Why also a runtime hook.** The TypeBox schema catch is compile-time-ish (per-run). The `before_tool_call` hook is the runtime backstop in case the schema cache is stale or a tool was registered with a static enum (covering all states). Belt and suspenders.

**Multi-state surveys.** When the TaskSpec includes multiple jurisdictions, R2 permits any tool call whose jurisdiction is in the list. W1 PartialAcrossStates then watches for per-state coverage: if the agent retrieves CA statutes but never IL statutes despite both being in scope, W1 surfaces the partial-coverage warning at delivery.

**Sub-jurisdictions.** County, city, or agency-level jurisdictions (e.g., LA County's healthcare-worker rules) are first-class. The format `"US-CA:los-angeles-county"` is interpreted by the corpus layer; R2 matches at the prefix level (a request for `"US-CA"` succeeds when the TaskSpec has `"US-CA:los-angeles-county"` because the city is a subset of the state, but not vice versa). This is encoded in the comparator function `isJurisdictionInScope(requested, allowed[])` in the plugin pack.

## Interactions with other rules

- **R1 SourceAllowlist** fires at the same hook (`before_tool_call`). R1 controls *which source* is reachable; R2 controls *which jurisdiction* within that source is addressable. Both must pass.
- **R15 CompletenessGate** depends on R2 to know what "complete" means: completeness is per-jurisdiction. A multi-state survey is complete only when every jurisdiction in TaskSpec.jurisdictions has passed its own R15 check.
- **W1 PartialAcrossStates** is the warning surfaced when R2 admits a jurisdiction but the agent fails to actually retrieve from it.

## Test cases

✅ Compliant: in-scope jurisdiction
```
TaskSpec: { jurisdictions: ["US-CA"], source_profile: "B" }
Tool call: kb_statutes_search({ jurisdiction: "US-CA", query: "independent contractor" })
Result: tool executes; results returned.
```

✅ Compliant: sub-jurisdiction under scoped state
```
TaskSpec: { jurisdictions: ["US-CA:los-angeles-county"], source_profile: "B" }
Tool call: kb_regs_search({ jurisdiction: "US-CA", query: "..." })
Result: tool executes — LA County is a subset of CA, so a CA-level query is in scope. (State-level queries match against the parent jurisdiction.)
```

❌ Non-compliant: out-of-scope jurisdiction
```
TaskSpec: { jurisdictions: ["US-CA"], source_profile: "B" }
Tool call: kb_statutes_search({ jurisdiction: "US-TX", query: "..." })
Result: { block: true, reason: "JURISDICTION_VALIDATION_ERROR: tool \"kb_statutes_search\" targets jurisdiction \"US-TX\" which is not in TaskSpec.jurisdictions [\"US-CA\"]. Either expand TaskSpec.jurisdictions at intake or pick a different tool." }
```

❌ Non-compliant: state-level query when only county is in scope
```
TaskSpec: { jurisdictions: ["US-CA:los-angeles-county"], source_profile: "B" }
Tool call: kb_statutes_search({ jurisdiction: "US-CA", query: "..." })  // state-wide is broader than county
Result: { block: true, reason: "JURISDICTION_VALIDATION_ERROR: requested \"US-CA\" (state-level) is broader than the locked sub-jurisdiction \"US-CA:los-angeles-county\". Either expand TaskSpec to include US-CA or narrow the tool call to the locked sub-jurisdiction." }
```

❌ Non-compliant: TaskSpec has no jurisdictions
```
TaskSpec: { jurisdictions: [], source_profile: "B" }   // intake bug
Tool call: kb_statutes_search({ jurisdiction: "US-CA", query: "..." })
Result: schema validation fails — TaskSpec.jurisdictions must be non-empty (TaskSpec schema constraint). R2 never sees this call because the TaskSpec itself is invalid. The orchestrator rejects the run at Tier-1 intake with "JURISDICTION_LOCK_REQUIRED: TaskSpec.jurisdictions cannot be empty."
```

## Open questions

- **Federal jurisdiction.** Should `"US"` (federal) be a permitted jurisdiction code, or must federal-law questions be expressed as a separate `federal_law: true` flag on the TaskSpec? Recommend: keep `"US"` as a permitted code so the same R2 machinery handles federal-state-county uniformly; revisit if federal-only questions become a common pattern.
- **Multi-state inheritance.** If TaskSpec.jurisdictions is `["US-CA", "US-IL", "US-NJ"]`, does R2 require each tool call to name one of these explicitly, or can a tool call omit the param and receive results across all three? Current spec: explicit per call. The agent must iterate.
- **Cross-jurisdiction comparison.** When the analysis legitimately compares two states (e.g., "California's ABC test vs. Illinois's"), R2 still requires each tool call to be single-jurisdiction. The orchestrator handles the cross-comparison logic; the tool layer stays narrow. This is intentional and not an open question — it's recorded here only to head off the "why can't I just query both" feature request.
