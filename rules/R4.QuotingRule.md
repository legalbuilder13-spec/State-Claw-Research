# R4.QuotingRule

**Spec:** Any text in quotation marks in any deliverable, claim, sub-agent artifact, or audit-log entry must carry a `(doc_id, char_start, char_end, hash)` anchor pointing to a span retrieved this session and present in the `retrieval_log` table. Unanchored quotation marks are rejected at three independent enforcement points (Drafter schema, Renderer schema, Critic exact-match), each capable of blocking delivery on its own.

**Enforcement point:**
1. **Drafter output schema** (`schemas/claim.schema.json`) — the Drafter cannot construct a claim whose `quotes[]` array contains an entry without a valid anchor; schema validation runs in the `before_agent_finalize` hook of the Drafter sub-agent and triggers a `revise` loop on violation.
2. **Renderer pre-render check** (`schemas/deliverable.schema.json`) — the Renderer scans every string field of the final deliverable for unwrapped quotation-mark characters; any unwrapped quote fails the render.
3. **Critic sub-agent** — independently re-fetches each anchored span via `kb_span_get_by_hash` and exact-substring-matches the quoted text against the fetched span. Any mismatch blocks delivery.

**Severity:** blocking.

**Failure mode:** The quoted span is rejected. The Drafter must rewrite as paraphrase (no quotation marks) or remove the assertion entirely. If a Drafter persistently produces unanchored quotes after 2 `before_agent_finalize` revise loops, the agent escalates to R11 LowConfidenceGate and the deliverable is blocked with "research incomplete." If the Critic finds an anchor that *exists* but whose hash doesn't exact-match the fetched span (typically because the corpus was re-ingested mid-session and the hash drifted), R10 CurrencyTag fires alongside R4 and the run is restarted with the fresh corpus.

## Detail

The Quoting Rule is the single most important rule in the system. A paraphrase converted to a quote is an attestation: "these are the exact words." Most legal hallucinations from LLMs are paraphrases that hardened into quotes during drafting — the model rephrased a statute, lost a key qualifier ("may" vs. "shall," "an" vs. "the," "of" vs. "for"), then committed the rephrasing to quotation marks because the surface-level structure looked authoritative. By the time a human reads the output, the quoted text reads like a statute citation; only the actual source document reveals the drift.

R4 makes this failure mode mechanically impossible:

1. The Drafter cannot author a quote without a structural anchor; the JSON Schema rejects it.
2. The Renderer cannot pass through a string with unwrapped quotation marks; the schema rejects it.
3. The Critic re-fetches and exact-matches every anchored quote against the live corpus; any drift between the asserted quote and the source span fails.

Three independent enforcement points, all blocking. A bug in any single layer is caught by the next layer.

**Anchor shape.** Every quote is wrapped in a structured object:

```json
{
  "text": "the contracting business demonstrates that the worker is",
  "anchor": {
    "doc_id": "ca-lab-2775-a-2",
    "char_start": 1023,
    "char_end": 1078,
    "hash": "sha256:7c8a..."
  }
}
```

- `doc_id` is the corpus identifier (`<source_category>:<jurisdiction>:<identifier>` per PRD §11 conventions).
- `char_start` / `char_end` are character offsets into the canonical text of `doc_id`.
- `hash` is the SHA-256 of the span at those offsets at the time of retrieval — this is what makes drift detectable.

The Renderer's final output renders quotes as plain text (e.g., `"the contracting business demonstrates that the worker is"`) but the rendered deliverable also contains a `cites[]` block carrying the anchor metadata for every quote, so reviewers can click through to the evidence card and verify.

**Paraphrasing is the escape hatch.** R4 does not prevent paraphrasing — it prevents the model from claiming a paraphrase is a quote. The Drafter is encouraged to paraphrase whenever the precise wording isn't legally load-bearing, *as long as* the paraphrase doesn't use quotation marks. The Critic separately checks for paraphrase fidelity via W2 ParaphraseDriftCheck.

**Boundary cases.** Quoting rules in legal writing have well-known boundary cases the rule must handle:
- *Block quotes* are still quotes. R4 applies.
- *Quoted citations within quoted text* (nested quotes via `'...'` inside `"..."`) are still quotes. R4 applies to the outermost quotation marks.
- *Section labels* like `"§ 2775(b)(1)(A)"` are not quotes (they're identifiers); R4's preprocessor strips these from the candidate-quote scan via a regex match against the citation taxonomy in `schemas/source-categories.json`.
- *Single-word emphasis* like `"shall"` vs. `shall` — R4 still applies. Even single quoted words are anchored. (This is the case where hallucination most commonly happens — a model emphasizes one word, picks the wrong one, and the reader trusts the quotation marks.)
- *Scare quotes* (`the so-called "independent" contractor`) — R4 still requires an anchor. The pragmatic fix is to use italics or angle brackets instead of quotation marks when scare-quoting; the JSON Schema permits both.

**Why the hash.** The hash is what makes R4 cheap to enforce. Without it, the Critic would have to do a substring search against the entire `retrieval_log` table for every quote. With it, the Critic does a single keyed lookup. A hash mismatch (rather than a substring miss) is also a much louder signal — it means either the corpus changed between Drafter time and Critic time (R10 CurrencyTag fires) or the Drafter fabricated the anchor (R5 HashEcho fires).

## Interactions with other rules

- **R3 NoMemoryStatutes** is the broader umbrella: R3 says every statutory assertion must trace to a retrieved span; R4 is the strict subset that says every *quoted* span must additionally carry an exact-match anchor. R3 fires for paraphrased assertions; R4 fires for quoted ones.
- **R5 HashEcho** is the Verifier-side check: the Verifier re-fetches every anchored span and confirms the hash. If R4 says "the Drafter must anchor," R5 says "the Verifier must independently confirm." Both must pass.
- **R6 PinpointCite** is the citation-format rule: R6 says every claim must carry a pinpoint cite (title + section + subdivision); R4 says every quote must carry a span anchor. These are different artifacts: a cite identifies *which document*; an anchor identifies *which characters of that document*. Both must be present for quoted spans.
- **R10 CurrencyTag** is the freshness check: an anchor's `hash` mismatching the live source typically means the corpus was re-ingested mid-session. R10 surfaces this; the agent then either restarts with the fresh corpus or annotates the quote with a "current_through" note.
- **W2 ParaphraseDriftCheck** is the cousin warning for non-quoted text: between two Drafter revisions, did a paraphrase drift from its anchored source? W2 doesn't block; it annotates.

## Test cases

✅ Compliant: anchored quote
```
Drafter emits a claim:
{
  "text": "Under California Labor Code § 2775, ",
  "quotes": [
    {
      "text": "a person providing labor or services for remuneration shall be considered an employee",
      "anchor": { "doc_id": "ca-lab-2775", "char_start": 0, "char_end": 90, "hash": "sha256:abcd..." }
    }
  ],
  "cites": [{ "kind": "primary_statute", "jurisdiction": "CA", "code": "Lab.", "section": "2775", "subdivision": "(a)" }]
}

R4 enforcement:
- Drafter schema: ✓ quote is wrapped with valid anchor
- Renderer schema: ✓ no unwrapped quotation marks in any string field
- Critic: fetches span via kb_span_get_by_hash("sha256:abcd..."); exact-matches; ✓
- Verifier: independently fetches; hash matches; ✓
Result: claim accepted; renders as the statute text in quotes with a click-through to the evidence card.
```

❌ Non-compliant: unanchored quote in Drafter output
```
Drafter emits:
{ "text": "the statute says \"the worker is presumed an employee\"" }   // unwrapped quotation marks in the text field

R4 enforcement:
- Drafter schema: ✗ raw quotation marks detected in a string field that doesn't have a matching quotes[] entry
Result: { revise: true, instructions: "R4: every quoted span must be moved to the quotes[] array with a valid anchor or rewritten as paraphrase. Detected unanchored quote: \"the worker is presumed an employee\"." }
```

❌ Non-compliant: paraphrase masquerading as quote
```
Drafter emits a claim citing CA Lab. § 2775(a). The actual statute reads:
"a person providing labor or services for remuneration shall be considered an employee"

But the Drafter writes (with anchor):
{
  "text": "...",
  "quotes": [{ "text": "the worker must be classified as an employee", "anchor": { "doc_id": "ca-lab-2775", "char_start": 0, "char_end": 90, "hash": "sha256:abcd..." } }]
}

R4 enforcement:
- Drafter schema: ✓ structurally valid
- Renderer schema: ✓ structurally valid
- Critic: fetches span; the fetched span text is "a person providing labor or services for remuneration shall be considered an employee" — does NOT exact-match "the worker must be classified as an employee".
Result: { block: true, reason: "R4: quoted text does not match anchored source span. Anchored doc_id=ca-lab-2775 chars 0-90 reads: \"a person providing labor or services for remuneration shall be considered an employee\". Drafter asserted: \"the worker must be classified as an employee\". This is paraphrase, not quote — rewrite without quotation marks or re-quote the actual span." }
```

❌ Non-compliant: forged anchor
```
Drafter emits a quote with an anchor whose hash doesn't match any entry in retrieval_log.

R4 enforcement:
- Drafter schema: ✓ (Drafter can write any hash it wants; the schema doesn't validate hash existence on its own)
- Renderer schema: ✓
- Critic: kb_span_get_by_hash("sha256:forged...") returns NOT_FOUND.
Result: { block: true, reason: "R4: anchored hash sha256:forged... not present in retrieval_log. Either the span was not retrieved this session (R3 violation) or the hash was fabricated (R5 HashEcho violation). Either way, blocked." }
```

❌ Non-compliant: corpus drift (hash mismatch on live re-fetch)
```
Drafter emits a quote with anchor doc_id=ca-lab-2775 hash=sha256:abcd... (retrieved early in the session).
Mid-session, the ingestion worker re-ingested CA Labor Code (e.g., new amendment).
Critic re-fetches and computes the live hash: sha256:zzz... (different).

R4 enforcement:
- Drafter schema: ✓
- Renderer schema: ✓
- Critic: hash mismatch.
Result: R4 fires AND R10 CurrencyTag fires. Orchestrator decision:
  (a) If the underlying span text is identical (whitespace/normalization drift only) → log W2 ParaphraseDriftCheck and allow.
  (b) If the text materially changed → restart the session with the fresh corpus (R10 path) and re-Draft.
```

✅ Compliant: paraphrase (no quotes)
```
Drafter emits a claim without quotation marks:
{ "text": "Under § 2775, a worker providing services for pay is treated as an employee unless the hiring entity satisfies the ABC test...", "cites": [{ "kind": "primary_statute", "jurisdiction": "CA", "code": "Lab.", "section": "2775" }] }

R4 enforcement: no quotation marks → R4 does not fire. (R3 NoMemoryStatutes still applies to the underlying assertion, enforced via the Critic comparing the paraphrase to the cited section.)
Result: accepted as paraphrase.
```

❌ Non-compliant: scare quotes without anchor
```
Drafter emits: "the so-called \"independent\" contractor"

R4 enforcement:
- Drafter schema: ✗ unwrapped quotation marks; scare-quote exception does not apply (no whitelist exists for it).
Result: { revise: true, instructions: "R4: scare quotes are still quotes. Either anchor \"independent\" to a span where the statute uses that exact word, rewrite with italics, or remove the emphasis." }
```

## Open questions

- **Section-label preprocessor.** R4's preprocessor strips citation tokens like `"§ 2775(b)(1)(A)"` from the candidate-quote scan so they aren't treated as quotes. The exact regex is in `schemas/source-categories.json`'s `citation_token_patterns` field (to be authored in Phase 1). Until that lands, R4's implementation uses a permissive fallback.
- **Block quote rendering.** Slack doesn't render markdown block-quote syntax (`>`) the same way as a memo; the Renderer needs distinct rendering paths per channel. R4 itself is channel-agnostic (it only cares about anchor presence), but the rendering pass must preserve the anchor link structure across channels.
- **Quote attribution for synthesized text.** If the Drafter combines two retrieved spans into a single sentence with quotation marks around the concatenation, R4 currently rejects (the concatenation has no single anchor). Allowing concatenation-with-multi-anchor is a Phase 2 enhancement if it turns out to be common; for v1, the Drafter must quote each span separately.
- **Non-English quote characters.** Curly quotes (`"…"`), angle quotes (`«…»`), and corner brackets (`「…」`) all denote quotation. R4's preprocessor normalizes these to straight ASCII quotes before scanning. The normalization map lives in the Drafter's preprocessor; extend it if new quote glyphs appear in retrieved corpora.
