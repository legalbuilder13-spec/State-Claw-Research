# R12.ClassificationOverlay

**Spec:** Every deliverable must include a classification-risk overlay: a structured block listing every category from `company-context/risk-taxonomy.yaml` with each category either **flagged** (a one-sentence rationale describing how the deliverable's subject implicates that risk category) or **explicitly cleared** (a one-sentence rationale describing why the risk category does not apply). Missing categories cause render failure.
**Enforcement point:** The Renderer's schema validation (`schemas/deliverable.schema.json`) requires every key in `risk-taxonomy.yaml` to appear in the deliverable's `classification_overlay` block with a status of `flagged | cleared` and a non-empty `rationale` field. Validation fires in the `before_agent_finalize` hook.
**Severity:** blocking.
**Failure mode:** Render fails with `CLASSIFICATION_OVERLAY_INCOMPLETE: missing category "<key>" in classification_overlay block; every category in company-context/risk-taxonomy.yaml must appear as either flagged (with rationale) or explicitly cleared (with rationale). The Classification sub-agent must complete the overlay before delivery.` The Classification sub-agent is invoked to fill missing entries; persistent failures escalate to R11.

## Detail

R12 is the rule that makes the company-context risk taxonomy operational in every output. Every legal-research deliverable carries an explicit, structured assessment of which company-specific risk categories the deliverable's subject matter touches. This converts the agent from a generic legal-research tool into a Clipboard-Health-specific (or any-operator-specific) risk-aware research tool.

**Why this is non-optional.** A research deliverable that doesn't surface classification risk is, for an in-house team, missing the point of using the system. The team isn't researching legal questions in the abstract — they're researching them because they have classification exposure (or licensing exposure, or joint-employer exposure, etc.) and need to know how the law applies to *their* fact pattern. R12 forces every deliverable to make that mapping explicit. The reviewer can disagree with the agent's classification, but they can't be left unaware that a question even *implicates* a category.

**Taxonomy as the source of truth.** The risk taxonomy lives at `company-context/risk-taxonomy.yaml` (Phase 1 deliverable). Default gig-economy categories per PRD §9 and the CBH context in this user's global CLAUDE.md:

```yaml
worker_classification:
  abc_test: { jurisdictions: [US-CA, US-MA, US-NJ, US-IL, US-MN, ...] }
  common_law_agency: { jurisdictions: [US, all_states] }
  joint_employer_flsa: { jurisdictions: [US] }
  joint_employer_state: { jurisdictions: [all_states] }
  marketplace_contractor_statutes: { jurisdictions: [...] }
  staffing_agency_statutes: { jurisdictions: [...] }

control_indicators:
  behavioral_control:    # supervision, direction, training
  financial_control:     # rate-setting, exclusivity, business-expense exposure
  relationship_indicators:  # written contracts, permanency, integration into business

licensing:
  staffing_agency_licensing: { jurisdictions: [...] }
  professional_employer_organization: { jurisdictions: [...] }

other:
  conversion_fee_prohibition:   # direct-hire / temp-to-perm fees
  hcf_policy_compliance:        # facility-policy compliance contractual language
  background_check_obligations: # who runs which checks
  hipaa_baa_required:           # PHI handling
```

The Classification sub-agent reads this YAML, generates one overlay entry per top-level key + nested key, and ensures every leaf has a status + rationale.

**Flagged vs. cleared.** A category is **flagged** when the deliverable's subject matter implicates the risk (the analysis touches behavior, language, or law in that category). A category is **cleared** when the deliverable's subject matter does not implicate the risk (the analysis is about something else). Both statuses require a one-sentence rationale; clearance without rationale is rejected. This forces the Classification sub-agent to actually think about each category rather than copy-paste boilerplate.

**Severity within the overlay.** Each flagged category carries a severity: `low | medium | high | critical`. Severity drives the Slack-response template's color (green / yellow / orange / red) and ordering (critical first). The Classification sub-agent's logic for severity assignment is in `skills/classification-overlay/SKILL.md` (Phase 1 deliverable).

**CBH terminology rules.** The user's global CLAUDE.md establishes hard terminology constraints for Clipboard Health: never call professionals "employees," never call CBH a "staffing agency" or "employer," etc. R12's rationale text is generated by an LLM, so the Classification sub-agent must enforce those constraints in its outputs. The skill file at `skills/classification-overlay/SKILL.md` includes the terminology rules as constraints; the Critic re-checks rationale text for violations. This is one of the load-bearing R12 enforcement points specifically tuned to CBH.

**Per-vertical overrides.** If the fork is reused for a non-CBH gig-economy vertical (rideshare, delivery, healthcare staffing, etc.), the operator edits `company-context/risk-taxonomy.yaml` to add/remove categories. R12's enforcement adapts automatically — it reads whatever is in the YAML. PRD §2 lists the verticals this is designed to support.

## Interactions with other rules

- **R14 NoLawFirmCitation** is the source-category-side counterpart: R12 governs the company-context-overlay block; R14 governs the cites block. Independent.
- **R6 PinpointCite** is required for any cite referenced inside an overlay rationale. If the Classification sub-agent quotes a statute in a rationale, that quote needs R4/R6 compliance.
- **R11 LowConfidenceGate** consumes R12's completeness — a missing-categories overlay is a confidence failure that contributes to R11.
- **R15 CompletenessGate** has "classification overlay applied" as its 7th criterion; R12's pass is required for R15's pass.

## Test cases

✅ Compliant: full overlay with mixed statuses
```
Risk taxonomy has 8 categories. The deliverable's subject is California AB5 / ABC Test.
Classification overlay:
{
  abc_test:                       { status: "flagged",  severity: "critical", rationale: "Question is squarely about Prong B of the CA ABC test." }
  common_law_agency:              { status: "cleared",  rationale: "ABC test displaces common-law agency in CA; common-law analysis not required for this question." }
  joint_employer_flsa:            { status: "cleared",  rationale: "Question does not involve federal wage-and-hour exposure with a co-employer." }
  joint_employer_state:           { status: "flagged",  severity: "medium",   rationale: "CA's joint-employer doctrine could apply if the analysis concludes worker is an employee; secondary consideration." }
  marketplace_contractor_statutes:{ status: "cleared",  rationale: "CA has not enacted a marketplace-contractor statute; not applicable." }
  staffing_agency_statutes:       { status: "flagged",  severity: "high",     rationale: "Conclusion turns on whether CBH is characterized as a staffing agency under state licensing definitions; load-bearing for the analysis." }
  ...
}
Result: render passes; deliverable includes the overlay block prominently above the analysis.
```

❌ Non-compliant: missing category
```
Risk taxonomy has 8 categories; classification_overlay has 7.
Render: { block: true, reason: "R12: CLASSIFICATION_OVERLAY_INCOMPLETE: missing category \"hcf_policy_compliance\". Every category in risk-taxonomy.yaml must appear as flagged or explicitly cleared." }
Classification sub-agent is invoked to fill the missing entry.
```

❌ Non-compliant: cleared without rationale
```
classification_overlay.joint_employer_flsa: { status: "cleared", rationale: "" }
Render: { block: true, reason: "R12: cleared category requires non-empty rationale. Either provide a one-sentence rationale or change status to flagged." }
```

❌ Non-compliant: rationale uses banned terminology
```
classification_overlay.abc_test: { status: "flagged", severity: "critical", rationale: "Clipboard Health employs nurses through its platform..." }
Critic flags: rationale uses "employs nurses" — violates terminology rule "never describe CBH as employing professionals."
Result: revise loop. Classification sub-agent re-emits with corrected language: "Clipboard Health facilitates marketplace bookings of independent healthcare professionals; ABC test applies because..."
```

✅ Compliant: severity-based ordering in render
```
Two flagged categories: abc_test (critical), joint_employer_flsa (low).
Renderer sorts flagged entries by severity descending: abc_test first, joint_employer_flsa second.
Slack render uses 🔴 emoji for critical, 🟡 for low (template-defined).
```

## Open questions

- **Auto-clearance heuristics.** Some categories should always clear for some subject matters (e.g., HIPAA-BAA for an IC-classification analysis with no PHI implication). Should the Classification sub-agent get a starter clearance heuristic table to reduce its workload? Recommend yes — Phase 1 deliverable as part of `skills/classification-overlay/SKILL.md`.
- **Severity calibration across deliverables.** Two different deliverables that touch the same risk category should produce similar severity ratings. Calibration drift over many deliverables is a real concern. Mitigation: log severity assignments and re-evaluate quarterly against actual outcomes.
- **Cross-deliverable persistence.** When the same risk category is flagged on multiple deliverables for the same matter (e.g., a multi-state survey), the team should see the consolidated risk picture across all deliverables. This is a Phase 6+ matter-management feature; for v1, each deliverable carries its own self-contained overlay.
- **Per-jurisdiction category gating.** Some risk categories are jurisdiction-conditional (e.g., ABC test applies in CA, MA, NJ, IL, MN, NM; common-law in others). The risk-taxonomy.yaml has per-category `jurisdictions` fields; the Classification sub-agent should auto-clear categories where the deliverable's jurisdictions don't intersect with the category's jurisdictions. This logic lives in the skill.
