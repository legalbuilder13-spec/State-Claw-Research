# R6.PinpointCite

**Spec:** Every `cites[]` entry in the final deliverable must carry pinpoint identifiers sufficient to locate the cited material to a single provision: for statutes, `title + section + subdivision`; for regulations, `title + part + section + subdivision`; for cases, `case name + reporter + volume + page + court + year + pinpoint page`; for treatises, `author + title + edition + page or section`. Citations with missing pinpoint fields are rejected at render time.
**Enforcement point:** Renderer output schema (`schemas/deliverable.schema.json`) validates every entry in `cites[]` against a discriminated union of citation shapes keyed on `kind`. Missing pinpoint fields fail schema validation and block delivery via the `message_sending` hook.
**Severity:** blocking.
**Failure mode:** Render fails with the error text `PINPOINT_CITATION_REQUIRED: cite #<index> of kind "<kind>" is missing required pinpoint field(s): [<fields>]. Either complete the citation from the retrieval log or remove the claim.` The Drafter is invoked again with a `revise` action listing the incomplete citations. Persistent failures after 2 revise loops escalate to R11.

## Detail

A "pinpoint" citation is the citation form an attorney would use in a brief — specific enough that a reader can find the exact paragraph the citation supports. The opposite is a "string cite" or "name only" — `Cal. Lab. Code § 2775` without a subdivision, or `Dynamex` without a reporter — which is useful for orientation but doesn't let the reader verify the assertion. R6 enforces pinpoint precision because the rest of the system depends on it: R5 HashEcho needs the exact doc_id to re-fetch; the audit chain needs the exact provision to log; the reader needs the exact reference to verify.

**Citation shape by kind.** The schema enforces different required fields per `kind`:

```yaml
primary_statute:
  required: [jurisdiction, code, section]
  recommended: [subdivision, paragraph, year_enacted]
  example: { jurisdiction: "US-CA", code: "Lab.", section: "2775", subdivision: "(b)(1)(A)" }
  renders_as: "Cal. Lab. Code § 2775(b)(1)(A) (2019)"

primary_regulation:
  required: [jurisdiction, title, part, section]
  recommended: [subdivision, paragraph]
  example: { jurisdiction: "US", title: "29", part: "541", section: "100", subdivision: "(a)" }
  renders_as: "29 C.F.R. § 541.100(a)"

case_law:
  required: [name, reporter, volume, first_page, court, year]
  recommended: [pinpoint_page, parenthetical]
  example: { name: "Dynamex Operations W. v. Superior Court", reporter: "Cal. 5th", volume: "4", first_page: "903", pinpoint_page: "957", court: "Cal.", year: "2018" }
  renders_as: "Dynamex Operations W. v. Superior Court, 4 Cal. 5th 903, 957 (2018)"

secondary_treatise:
  required: [author, title, edition_or_year]
  recommended: [section_or_page]
  example: { author: "Restatement (Third) of Employment Law", section_or_page: "§ 1.01" }
  renders_as: "Restatement (Third) of Employment Law § 1.01"
```

The full schema is in `schemas/citation.schema.json`. Citations missing required fields fail validation at render time. Recommended fields are not required — but their absence triggers W3 HedgeDensity-like soft warnings if they're commonly available and missing.

**Why subdivision matters.** Most statutory hallucinations from LLMs cite the right section but the wrong subdivision. "Section 2775 says X" might be true at the section level but false at subdivision (b)(2). The pinpoint requirement forces the agent to commit to a specific provision, which then either matches the retrieval log entry (R3) and the anchor (R4/R5) or fails one of them. Without the subdivision, those downstream checks have nothing to verify against.

**Case pinpoint pages.** For cases, the pinpoint page (the specific page within the opinion where the cited proposition appears) is *recommended* not required, because some uses of a case (a string cite of authority, a `See generally` reference) legitimately don't have a pinpoint. But for any *holding* the deliverable relies on, the pinpoint page is required — this is enforced by a tighter check: if the cite is referenced in a claim with `relies_on_holding: true` (set by the Drafter when the claim's force comes from the case's specific reasoning), then `pinpoint_page` is required.

**Bluebook style by default.** The rendered citation form uses Bluebook (per PRD §20 decision 10 default). Other styles (ALWD, jurisdictional preferences) are configurable per company-context — see `company-context/citation-style.yaml` (added in Phase 1 company-context work). The Renderer reads this config and switches the rendering layer accordingly; the underlying schema is style-agnostic.

**Cross-jurisdictional consistency.** Federal regulations cited in a state-law deliverable still need the federal jurisdiction code (`"US"`) plus the federal C.F.R. structure (title + part + section). A state regulation cited in a federal-law deliverable still needs the state jurisdiction code plus state CCR structure. The schema enforces this — there's no implicit "default jurisdiction" — which prevents the subtle bug where the agent says "29 C.F.R. § 541" without indicating it's federal.

## Interactions with other rules

- **R3 NoMemoryStatutes** is the existence check (the cited doc_id must be in `retrieval_log`); R6 is the form check (the cite must have the pinpoint fields). Both must pass.
- **R4 QuotingRule** anchors *spans* to *characters*; R6 anchors *claims* to *citations*. A quoted claim has both an anchor (R4) and a cite (R6).
- **R5 HashEcho** verifies the anchor matches the source; the cite's pinpoint fields are required for the Renderer to display the verifiable link.
- **R14 NoLawFirmCitation** is the kind check (only `citation_eligible` kinds in `cites[]`); R6 is the field check (whatever kind is used, it must be pinpoint-complete).

## Test cases

✅ Compliant: full pinpoint statute cite
```
Cite: { kind: "primary_statute", jurisdiction: "US-CA", code: "Lab.", section: "2775", subdivision: "(b)(1)(A)" }
Renders as: "Cal. Lab. Code § 2775(b)(1)(A)"
Result: schema passes; deliverable proceeds.
```

✅ Compliant: case with pinpoint page (for a holding-reliant claim)
```
Claim: { text: "...", relies_on_holding: true, cites: [{ kind: "case_law", name: "Dynamex Operations W. v. Superior Court", reporter: "Cal. 5th", volume: "4", first_page: "903", pinpoint_page: "957", court: "Cal.", year: "2018" }] }
Renders as: "Dynamex Operations W. v. Superior Court, 4 Cal. 5th 903, 957 (2018)"
Result: schema passes.
```

❌ Non-compliant: statute cite missing subdivision when needed
```
Drafter asserts: "Under section 2775(b), the employer must establish..." with cite: { kind: "primary_statute", jurisdiction: "US-CA", code: "Lab.", section: "2775" }   // subdivision missing
Result: schema rejects because the asserted text references "(b)" but the cite lacks the subdivision field. { revise: true, instructions: "R6: cite #1 references subdivision (b) in the asserted text but cite.subdivision is missing. Either complete the cite or remove the subdivision reference from the assertion." }
```

❌ Non-compliant: case cite missing pinpoint page when relying on holding
```
Claim: { text: "Dynamex held that...", relies_on_holding: true, cites: [{ kind: "case_law", name: "Dynamex", reporter: "Cal. 5th", volume: "4", first_page: "903", court: "Cal.", year: "2018" }] }   // pinpoint_page missing
Result: schema rejects. { revise: true, instructions: "R6: claim relies on holding but cite.pinpoint_page is missing. Locate the specific page where the holding is stated and complete the cite, or remove the relies_on_holding flag (e.g., for a string cite)." }
```

❌ Non-compliant: regulation cite missing title
```
Cite: { kind: "primary_regulation", jurisdiction: "US", part: "541", section: "100" }   // title missing
Result: schema rejects. { revise: true, instructions: "R6: primary_regulation cite missing required field 'title'. CFR cites must include the title number (e.g., 29 for Labor, 26 for Tax)." }
```

❌ Non-compliant: missing jurisdiction
```
Cite: { kind: "primary_statute", code: "Lab.", section: "2775", subdivision: "(b)" }   // jurisdiction missing
Result: schema rejects. { revise: true, instructions: "R6: cite missing required field 'jurisdiction'. Add the jurisdiction code (e.g., US-CA for California Labor Code)." }
```

✅ Compliant: orientation reference (not a cite)
```
Drafter mentions a law firm blog inline ("Smith & Jones noted in a recent blog post that..."). The blog appears in `references[]` (orientation references) but NOT in `cites[]`.
Result: schema passes — R6 only governs `cites[]`. R14 separately requires that orientation sources never appear in `cites[]`, which is satisfied here.
```

## Open questions

- **Citation style switching.** Bluebook is the default. ALWD and jurisdictional styles (e.g., the California Style Manual for California state-court briefs) are configurable. Define the precedence order when a multi-jurisdictional deliverable involves multiple style preferences — likely: TaskSpec.citation_style if set; else company-context default; else Bluebook. To be specified in `company-context/citation-style.yaml`.
- **Short forms (id., supra, infra).** Bluebook permits short-form references after a full cite. For v1, the Renderer always emits full cites (no shortening). Short forms are a Phase 6 polish item.
- **Parallel citations.** A California Supreme Court case has parallel citations in Cal., Cal. Rptr., and (sometimes) S.W.2d/N.E.2d. The Renderer emits the official reporter by default; if `parallel: true` is set on the cite, all available parallels are emitted. Phase 4 implementation decision.
