# R10.CurrencyTag

**Spec:** Every retrieved span carries `current_through` metadata indicating the date through which the source content is known to be current. The Verifier sub-agent's status pass marks any span whose `current_through` is older than the configurable staleness threshold (default: 90 days for statutes, 30 days for regulations, 7 days for case-law citator results, 1 day for active-litigation flag lookups). Stale spans are either refreshed before the analysis closes or surfaced to the deliverable with explicit "current through" annotations.
**Enforcement point:** The Verifier sub-agent computes per-span staleness during its anchor-validation pass (alongside R5 HashEcho); stale spans are reported with status `stale` and the orchestrator decides between (a) refresh-and-redraft, (b) accept-and-annotate, depending on a freshness policy keyed on the source kind and the question's sensitivity.
**Severity:** blocking when the staleness threshold is exceeded and the question's sensitivity flag requires fresh data; warning (deliverable annotated) otherwise.
**Failure mode:** If a span is stale beyond threshold AND the question is flagged `requires_fresh: true` in the TaskSpec (e.g., "is this proposed bill currently in effect?"), the orchestrator triggers a corpus refresh for that jurisdiction and re-Drafts after fresh retrieval. If `requires_fresh: false` (most questions), the deliverable proceeds with a "current through" annotation prominently displayed (e.g., `⚠️ Cal. Lab. Code § 2775 current through 2024-12-15; subsequent amendments not reviewed`).

## Detail

R10 addresses the time dimension of legal research. Statutes get amended, regulations get superseded, and cases get overruled — and the agent's corpus has a freshness lag (the gap between when a source publishes a change and when the ingestion worker pulls it). A correct answer based on stale corpus content is still a wrong answer if the law has changed since. R10 makes the freshness gap visible rather than letting the agent silently produce stale conclusions.

**Per-source thresholds.** Different sources have different freshness expectations:

- **Statutes**: amend infrequently; 90-day default lag is acceptable for most questions. Active-legislation flags (questions tied to currently-pending bills) override this.
- **Regulations**: amend more frequently; 30-day default. Federal regs tracked via the daily Federal Register feed have a 7-day threshold.
- **Case law**: subsequent treatment can negate a case overnight (overruling, depublication, repudiation). The citator chain (Profile B: CourtListener + Google Scholar + CAP; Profile A: Lexis Shepardize) is queried per-case with a 7-day threshold.
- **Agency guidance** (orientation): 14-day threshold; surfaced for context only (R14 prevents citation).

Thresholds are configurable in `extensions/agentic-law-os/r10-config.json` and overridable per TaskSpec via `freshness_overrides`.

**Per-question sensitivity flag.** Most legal research is about doctrines that don't change weekly — "what does the ABC test require?" is the same answer this week and next week. A 60-day-old retrieval is fine. But questions like "is this bill currently in effect?" or "did the Supreme Court grant cert on this issue?" require fresh data — the answer is invalidated by recent events. The TaskSpec carries a `requires_fresh: boolean` flag set at Tier-1 intake; when true, R10's threshold is tightened to the source-specific minimum.

**The corpus_freshness table.** The corpus tracks per-jurisdiction-and-source-kind ingestion timestamps in the `corpus_freshness` table (`migrations/0001_init.sql`). Every retrieved span carries the `current_through` value from this table at fetch time. R10's staleness computation is `now() - current_through`.

**Refresh and re-Draft.** When R10 fires with `requires_fresh: true` and the corpus is stale, the orchestrator:

1. Triggers a synchronous re-ingestion for the affected jurisdiction-and-source (`agentic-law-os-ingestion ingest-statutes --jurisdiction US-CA --force-refresh`). This blocks for up to 10 minutes (CA statutes ingest in 2–5 min typically).
2. After refresh, re-runs the affected retrievals.
3. Re-Drafts with fresh content.
4. Logs the refresh event to the audit chain with `R10_refresh: { source, jurisdiction, before, after }`.

If the refresh fails (state-site scraper broken, etc.), R10 escalates to R11 LowConfidenceGate with reason "could not confirm currency."

**Annotated delivery.** When `requires_fresh: false`, the deliverable proceeds but every cite displays its `current_through` prominently. The Slack template at `company-context/deliverable-templates/slack-response.md` (Phase 1 deliverable) includes a `> Sources current through:` callout listing the freshness of each source consulted. This is what the human reviewer sees.

**Negative treatment for cases.** For case citations, R10 runs the free citator chain (R10 + the `citator-profile-b` skill) — CourtListener "How Cited" + Google Scholar + CAP. Any negative treatment (overruled, abrogated, distinguished by named cases) is flagged. This is best-effort, not litigation-grade Shepardizing (PRD §3 non-goal 6); the deliverable annotates with "best-effort citator check; for litigation use, retain a human-led Shepardizing process."

## Interactions with other rules

- **R5 HashEcho** computes the live hash of every anchored span; R10 piggybacks on that fetch to also check `current_through`. Both run in the Verifier sub-agent.
- **R3 NoMemoryStatutes** depends on retrieval_log; R10 depends on the `current_through` metadata of those retrievals.
- **R15 CompletenessGate** treats unresolved staleness as a completeness failure (criterion 6 of 7).
- **R11 LowConfidenceGate** is the escalation path when refresh fails or staleness can't be confirmed.

## Test cases

✅ Compliant: span within freshness threshold
```
Span: ca-lab-2775, current_through: 2026-04-01 (40 days ago); statute threshold: 90 days.
R10: 40 < 90 → status: fresh. Deliverable proceeds; cite annotated "current through 2026-04-01".
```

⚠️ Conditional: span stale, requires_fresh false
```
Span: ca-lab-2775, current_through: 2025-12-15 (147 days ago); threshold: 90 days. TaskSpec.requires_fresh: false.
R10: stale → warning. Deliverable proceeds. Cite annotated "⚠️ current through 2025-12-15; subsequent amendments not reviewed (147 days old, exceeds 90-day threshold)."
Reviewer sees the warning prominently and decides whether to manually refresh.
```

❌ Non-compliant: span stale, requires_fresh true, refresh succeeds
```
TaskSpec.requires_fresh: true (question: "is AB1234 currently law?"). Span: stale.
R10: triggers `agentic-law-os-ingestion ingest-statutes --jurisdiction US-CA --force-refresh`.
Refresh completes in 3 min; corpus updated. Affected retrievals re-fetched. Drafter re-Drafts with fresh content.
Audit log: { R10_refresh: { source: "local_corpus_state_official", jurisdiction: "US-CA", before: "2025-12-15", after: "2026-05-11" } }
Result: deliverable produced from fresh content.
```

❌ Non-compliant: refresh fails
```
TaskSpec.requires_fresh: true. Span: stale. Refresh attempt: state-site scraper error.
R10: refresh failed; escalates to R11.
R11: blocks delivery; agent emits "research incomplete: could not confirm currency of CA statute corpus due to refresh failure. Last successful ingestion: 2025-12-15. Manual refresh recommended before re-running."
```

⚠️ Case-law negative treatment
```
Drafter cites Dynamex (2018). R10's case-specific check queries the citator chain.
CourtListener: 12 citations, all positive.
Google Scholar: no negative-treatment markers.
CAP: opinion still in force.
Result: status fresh; no negative treatment found. (For comparison, a case overruled by a subsequent decision would produce: status: "overruled_by: <case>"; deliverable annotated; recommend re-research.)
```

## Open questions

- **Refresh latency.** A synchronous mid-session refresh for a large jurisdiction (Texas Government Code is many thousands of sections) could exceed the 2-minute deliverable latency budget. For those cases, the orchestrator should fall back to "accept stale + annotate prominently" and surface the refresh as a recommended-next-step. Threshold for this fallback: configurable, default 10 minutes.
- **Threshold tuning.** The 90/30/7-day thresholds are guesses. Real-world data on legal-change frequency in target jurisdictions should refine. CA Labor Code amendments cluster in legislative session ending dates; the threshold could be session-aware (e.g., tighter in the 30 days after legislative session ends).
- **Citator chain confidence.** Free citators (CourtListener + Google Scholar) have lower recall than Shepardize/KeyCite. R10's negative-treatment check is therefore best-effort. The deliverable always prominently states this limitation. Profile A (Lexis) replaces the free chain with Shepardize when active.
- **Anti-staleness UX.** Should the agent proactively re-ingest stale jurisdictions on a schedule, or wait for the first stale-hit to trigger? Phase 2 design decision: schedule-based proactive refresh (cron in the ingestion worker) for high-traffic jurisdictions; on-demand for the long tail.
