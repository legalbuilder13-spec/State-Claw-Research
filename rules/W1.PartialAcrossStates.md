# W1.PartialAcrossStates

**Spec:** When a multi-state survey (TaskSpec.jurisdictions.length > 1) returns incomplete coverage of named jurisdictions — meaning at least one named state has fewer than the per-state minimum retrievals (default: 1 statute + 1 implementing-reg-or-skip-with-reason) — the deliverable surfaces a structured per-state coverage matrix as a top-level block, annotating which states were fully researched vs. partially researched vs. failed.
**Enforcement point:** The orchestrator runs the per-state completeness check after sub-agent dispatch completes (in the same pass as R15). Coverage matrix is added to the deliverable as a `coverage_matrix` block. Schema validation in `before_agent_finalize` requires the matrix to be present when `TaskSpec.jurisdictions.length > 1`.
**Severity:** warning.
**Failure mode:** Deliverable is annotated with the coverage matrix; it is NOT blocked. The matrix appears prominently above the analysis (per the Slack template) with color-coded states (green = complete, yellow = partial, red = failed). The reviewer sees explicit acknowledgment of partial coverage rather than a silent omission.

## Detail

W1 is the warning that prevents the multi-state survey from quietly omitting states. A reviewer asking "what do CA, IL, NJ, MA, and MN require for marketplace-contractor classification?" expects either an answer per state or an explicit "I couldn't cover IL because [reason]." The dangerous failure mode is a deliverable that covers CA, IL, NJ, MA, MN — but only really *researched* CA and IL, leaving the reviewer with the impression that all five were treated equivalently.

W1 makes this impossible by requiring an explicit per-state coverage matrix on every multi-state deliverable. Each state in TaskSpec.jurisdictions appears in the matrix with one of three statuses:

- **complete** — the state has at least 1 operative statute retrieved AND (any flagged delegation has a resolved implementing reg OR is explicitly marked `delegation_unresolved`) AND R15's per-state criteria all pass.
- **partial** — at least 1 operative statute retrieved, but some R15 criterion failed for this state (e.g., a cross-ref not traced, or a delegation reg not found).
- **failed** — fewer than 1 operative statute retrieved (the agent didn't actually research this state).

**Per-state R15.** R15's 7-criterion check is naturally per-claim, which means it can be evaluated per-state in a multi-state survey. The orchestrator iterates: for each state in TaskSpec.jurisdictions, run R15 against the claims tagged with that jurisdiction. The result feeds W1's matrix.

**The matrix's visual form.** The Slack template renders the matrix as a compact table:

```
Coverage:
✓ US-CA: complete (5 retrievals, all R15 green)
⚠️ US-IL: partial (3 retrievals; cross-ref to 820 ILCS 175 § 35 not traced)
✗ US-NJ: failed (0 operative retrievals; corpus does not have NJ ingested)
✓ US-MA: complete (4 retrievals)
✓ US-MN: complete (3 retrievals)
```

In a memo or chart deliverable (Phase 6), the matrix is a richer table with per-state research notes.

**No silent omission.** The schema requires every state in `TaskSpec.jurisdictions` to appear in `coverage_matrix`. If a state was named at intake but completely ignored by the analysis, the schema fails. This is what prevents the "I'll just focus on the easy states" failure mode.

**Why this is a warning, not blocking.** A partial survey is often still useful — the reviewer can ask "OK, please re-run for the failed states" or accept the partial coverage. Blocking the partial survey would be over-strict; surfacing it explicitly is sufficient. The cost of a partial survey delivered with a clear warning is much lower than the cost of a partial survey delivered without one.

**Interaction with R10 currency.** Each row in the coverage matrix includes the `current_through` date for that state's corpus. A reviewer can see at a glance: "OK, CA is current through 2026-04-01; IL is current through 2025-11-15 — IL might be stale."

## Interactions with other rules

- **R15 CompletenessGate** is the per-claim check; W1 is the per-state rollup. W1 evaluates after R15 (W1 needs to see R15's pass/fail per state).
- **R2 JurisdictionLock** ensures every state in scope is named in TaskSpec; W1 ensures those named states are actually researched.
- **R11 LowConfidenceGate** can fire alongside W1 if the failed/partial coverage drives an aggregate LOW confidence; in that case R11's incomplete-response replaces the deliverable, and W1's matrix appears within that response.

## Test cases

✅ Compliant: full 5-state coverage
```
TaskSpec.jurisdictions: [US-CA, US-IL, US-NJ, US-MA, US-MN]
All 5 states have ≥1 operative retrieval + R15 all-green per state.
coverage_matrix:
  US-CA: { status: "complete", retrievals: 5 }
  US-IL: { status: "complete", retrievals: 3 }
  US-NJ: { status: "complete", retrievals: 2 }
  US-MA: { status: "complete", retrievals: 4 }
  US-MN: { status: "complete", retrievals: 3 }
W1: no warning (matrix is uniformly complete). Deliverable proceeds.
```

⚠️ Compliant: partial coverage, matrix surfaced
```
TaskSpec.jurisdictions: [US-CA, US-IL, US-NJ]
US-CA: complete. US-IL: 1 unresolved cross-ref. US-NJ: 0 retrievals (NJ corpus not ingested).
coverage_matrix:
  US-CA: { status: "complete", retrievals: 5 }
  US-IL: { status: "partial", retrievals: 3, gaps: ["1 unresolved cross-ref to 820 ILCS 175 § 35"] }
  US-NJ: { status: "failed", retrievals: 0, reason: "NJ corpus not ingested; run agentic-law-os-ingestion ingest-statutes --jurisdiction US-NJ" }
W1: warning fires. Deliverable proceeds; Slack response prominently shows the matrix at the top.
```

❌ Non-compliant: missing state from matrix
```
TaskSpec.jurisdictions: [US-CA, US-IL, US-NJ]
coverage_matrix: has only 2 entries (CA, IL — NJ omitted).
Schema fails in before_agent_finalize: { revise: true, instructions: "W1: coverage_matrix is missing entry for US-NJ. Every state in TaskSpec.jurisdictions must appear in the matrix. Either add the entry (likely status: failed) or remove US-NJ from TaskSpec." }
```

❌ Non-compliant: matrix entry without required fields
```
coverage_matrix.US-IL: { status: "partial" }   // missing retrievals count and gaps[]
Schema fails: { revise: true, instructions: "W1: matrix entry for US-IL must include retrievals (number) and (if partial or failed) gaps[] with one entry per missed criterion." }
```

✅ Compliant: single-state, no W1 needed
```
TaskSpec.jurisdictions: [US-CA]
W1 does not fire (single-state). No coverage_matrix required.
```

## Open questions

- **Per-state minimum retrieval threshold.** Default: 1 statute + reg-or-skip. Some questions legitimately need only 1 statute (no delegation involved); others need 5+ statutes. Tunable per-deliverable from the topic taxonomy? Phase 4 design.
- **Cross-state comparison.** When the deliverable's purpose is to compare states (e.g., "which states require X-licensing"), W1's matrix is also the comparison table. Should the matrix's columns be question-specific (e.g., for X-licensing: licensure-required Y/N, threshold, exemptions)? Phase 6 chart-deliverable enhancement.
- **Auto-ingest on failed.** If W1 detects a "failed" state because the corpus isn't ingested, should it auto-trigger ingestion and re-run? Cost concern: ingestion can take minutes. Recommend asking the reviewer first; v1 surfaces the suggested command and waits.
- **Threshold for "partial" vs. "complete."** Currently any R15 failure makes a state "partial." Some R15 failures are minor (one stale cite); others are serious (unresolved exemption analysis). Worth subdividing partial into "minor" and "major"? Phase 4 design.
